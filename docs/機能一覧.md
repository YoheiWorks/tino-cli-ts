# **tino-cli の機能一覧**

- 題材は **Issue / Task / Note を扱うローカル管理ツール**
- 目的は **TS / Go の学習題材**
- ただのCRUDではなく、**設計・検証・テスト・改善**まで入れる
- 保存先はまず **ローカルファイル**
- 後で **Web API化** しやすい構造を意識する

です。

---

# tino-cli の位置づけ

`tino-cli` は、単なる todo ツールではなく、

**Issue / Task / Note をローカルで管理するワークスペースCLI** です。

扱うものは大きく3種類です。

- **task**: 実行する作業
- **issue**: 問題・不具合・対応事項
- **note**: メモ・補足・下書き

---

# 1. 基本機能

## item の作成

新しい item を追加する。

### 対象

- task
- issue
- note

### 例

- title を付けて作成
- description を任意で付ける
- tags を付ける
- project に所属させる

### コマンドイメージ

- `tino-cli item add`
- `tino-cli task add`
- `tino-cli issue add`
- `tino-cli note add`

---

## item の一覧表示

登録済み item を一覧で見る。

### 表示したいもの

- id
- type
- title
- status
- priority
- updatedAt
- project
- tags

### 形式

- text
- table
- json

### コマンドイメージ

- `tino-cli item list`

---

## item の詳細表示

1件の item を詳しく見る。

### 表示内容

- title
- description
- status
- priority
- tags
- createdAt
- updatedAt
- completedAt
- project

### コマンドイメージ

- `tino-cli item get --id ...`

---

## item の更新

既存 item の内容を更新する。

### 更新対象の例

- title
- description
- tags
- priority
- dueDate
- project

### コマンドイメージ

- `tino-cli item update --id ...`

---

## item の削除

不要な item を削除する。

### コマンドイメージ

- `tino-cli item delete --id ...`

---

# 2. 状態管理機能

これは `task` と `issue` が中心です。

## 完了

item を完了状態にする。

### やること

- status を `done` にする
- completedAt を入れる

### コマンドイメージ

- `tino-cli item complete --id ...`

---

## 再開

完了済み item を再開する。

### やること

- status を `done` 以外へ戻す
- completedAt を外す

### コマンドイメージ

- `tino-cli item reopen --id ...`

---

## status 更新

状態を変更する。

### 想定 status

- open
- in_progress
- blocked
- done

### コマンドイメージ

- `tino-cli item update-status --id ... --status blocked`

---

## 状態遷移ルール

ただ status を書き換えるだけではなく、ルールを持つ。

### 例

- `blocked -> done` は直接不可
- `done -> in_progress` は reopen 経由
- note には status を持たせない or 制限する

これはCLIの重要機能というより、**内部仕様**として重要です。

---

# 3. 分類・整理機能

## project 管理

item を project 単位でまとめる。

### project でできること

- 作成
- 一覧表示
- 更新
- アーカイブ

### コマンドイメージ

- `tino-cli project add`
- `tino-cli project list`
- `tino-cli project update`
- `tino-cli project archive`

---

## tags 管理

item に複数タグを付ける。

### できること

- add時にタグ付け
- updateでタグ変更
- tag で一覧絞り込み

### コマンドイメージ

- `tino-cli item add --tags backend,api`
- `tino-cli item list --tag backend`

---

## priority 管理

task / issue に優先度を付ける。

### 想定 priority

- low
- medium
- high

### コマンドイメージ

- `tino-cli item update --id ... --priority high`

---

## due date 管理

期限を持てるようにする。

### コマンドイメージ

- `tino-cli item update --id ... --due-date 2026-04-01`

---

# 4. 検索・絞り込み・並び替え

一覧を実用的にするための機能です。

## type フィルタ

- task だけ
- issue だけ
- note だけ

### 例

- `tino-cli item list --type task`

---

## status フィルタ

- open のみ
- done のみ
- blocked のみ

### 例

- `tino-cli item list --status open`

---

## project フィルタ

特定projectの item だけ見る。

### 例

- `tino-cli item list --project tino-api`

---

## tag フィルタ

タグ付き item を見る。

### 例

- `tino-cli item list --tag auth`

---

## キーワード検索

title / description を対象に検索する。

### 例

- `tino-cli item list --q login`

---

## 並び替え

一覧の順番を変える。

### 例

- createdAt
- updatedAt
- priority
- dueDate

### コマンドイメージ

- `tino-cli item list --sort updatedAt`

---

## ページング相当

CLIなので必須ではないけど、件数制限や先頭N件はあってもいい。

### 例

- `-limit 20`

---

# 5. 入出力機能

## import

外部JSONから item を取り込む。

### 目的

- seedデータ投入
- バックアップ復元
- 他環境から移す

### コマンドイメージ

- `tino-cli item import --file items.json`

---

## export

現在のデータをJSONで書き出す。

### 目的

- バックアップ
- API編への橋渡し
- 他ツールで利用

### コマンドイメージ

- `tino-cli item export --file backup.json`

---

## 一覧出力形式切替

一覧を機械処理向けにする。

### 形式

- table
- json
- ndjson も候補

### 例

- `tino-cli item list --format json`

---

# 6. 設定機能

## config ファイル読込

CLIの設定をファイルから読む。

### 想定項目

- データ保存先
- デフォルト出力形式
- timezone
- verbose mode

### コマンドイメージ

- `tino-cli item list --config ./tino.config.json`

---

## 環境変数対応

必要なら一部設定を env でも扱う。

### 例

- データファイルパス
- log level

---

# 7. 出力機能

## human-friendly 出力

人間向けに読みやすい形で出す。

### 例

- table
- 見やすい1行表示
- 状態ごとの印

---

## machine-friendly 出力

他のCLIやスクリプトから扱いやすい形で出す。

### 例

- json
- ndjson

---

## error 出力

失敗時に意味あるメッセージを返す。

### 例

- item not found
- invalid transition
- invalid input
- invalid config
- broken data file

---

# 8. 品質・設計面の機能

これはユーザー機能というより、教材としての中身です。

## validation

入力値をちゃんと検証する。

### 例

- title 空文字不可
- type 必須
- priority の不正値不可
- dueDate の形式チェック

---

## エラー分類

失敗理由を区別する。

### 例

- ValidationError
- NotFoundError
- InvalidTransitionError
- ConfigError

---

## repository / service 分離

CLIから直接ファイル保存しない。

### 意味

- Web API化しやすくする
- テストしやすくする
- 設計を学べるようにする

---

## テスト

最低限のUTを書く。

### 対象候補

- status遷移
- filter/search/sort
- service
- validation

---
